Spark的Shuffle过程原理剖析及源码分析

什么情况下会发生shuffle

在如下这些算子时会发生shuffle动作：reduceByKey、groupByKey、sortByKey、countByKey、join、corgroup等。总体来讲就是byKey类算子，join类算子。

默认情况下的shuffle原理剖析

默认情况下，Spark的shuffle原理图如下，如果该节点有2个cpu可用，三个shuffleMapTask：

每一个ShuffleMapTask都会为每一个ResultTask创建一份bucket缓存及ShuffleBlockFile磁盘文件。ShuffleMapTask的输出封装成MapStatus，发送到DagScheduler的MapOutputTrackerMaster中，其中包含了每个ResultTask要拉取的数据的大小。每个ResultTask会用BlockStoreShuffleFetcher去MapOutputTrackerMaster拉取自己要拉取的文件信息，然后底层通过BlockerManger将数据拉取过来，每个ResultTask拉取过来的数据就会组成一个内部的RDD，叫ShuffleRDD，优先放入内存，如果内存不够，则写入磁盘，然后每个RDD进行聚合形成新的MapPartitionsRDD。
如果一个节点上有100个maptask，100个resulttask，则会在本地产生10000个文件，磁盘IO过多性能下降。
第一个特点，在spark早起版本中，那个bucket的缓存非常重要，因为需要将一个ShuffleMapTask的所有数据都写入缓存后，才会刷新到磁盘，如果task的数据过多，那么很容易造成数据内存溢出，所以，新版本中默认的内存是100kb，若写入数据达到刷新磁盘的阈值时，就会将数据一点一点刷入磁盘。这种操作的特点是，不容易发生内存溢出。缺点在于如果内存缓存过小则会产生大量的IO操作，所以这里的内存缓存是可以根据实际情况进行优化的。
第二个特点，与MR不同，MR是必须将所有数据都写入本地磁盘后，才能启动MR操作，因为MR要实现默认的排序。但是Spark不需要，默认情况下，Spark是不需要进行排序的。因此maptask每写入一点数据，resulttask就可以拉取一点数据。 这种特点的好处就是比MR快速。但是，MR提供的reduce可以直接处理每个key上的value很方便。而Spark由于实时拉取的机制，因此无法直接处理key对应的values的算子，只能通过groupBykey先shuffle然后使用map算子进行value处理，没有MR方便。

优化后的shuffle原理剖析

新版本优化后，Spark的shuffle原理图如下，如果该节点有2个cpu可用，三个shuffleMapTask：

在spark的新版本中，引入了consolidation，也就是提出了ShuffleGroup的概念，一个节点上的一批并行的ShuffleMapTask执行完毕时，后续的ShuffleMaptask会复用之前的一批task的bucket的缓存，将数据写入对应的同一个文件中。每个磁盘文件包含多个ShuffleMapTask的数据（segment）此外会通过索引和偏移量进行不同task之间的数据区分，这个优化参数可以通过SparkConf的参数开启。如果一个节点上有2Core，有100个maptask，100个resulttask，开启consolidation后，会产生200个本地文件。
shuffle不会对每个分区的数据进行排序，如需要对shuffle后的数据进行排序，可以使用mappartition进行分区内的数据排序，不会出现shuffle。

BlockManger原理剖析与源码分析

Driver上有BlockMangerMaster，管理各个节点的BlockManger的内部的元数据。包含BlockMangerInfo，BlockerStatus
节点上有BlockManger，包含DiskStroe（负责磁盘数据读写），MemeryStroe（负责内存数据读写），BlockerTransferService（负责blockManger到其他节点BlockerManger的数据的读写），ConnectionManger（负责blockManger到其他节点BlockerManger的连接）。
每个BM创建之后，都会想BMM进行注册，此时BMM会为其创建对应的BMInfo。
BM进行写操作时，如RDD的中间数据，优先写入内存DS，内存不够时写入磁盘MS。
RDD的persist时，会使用CM同其他BM建立连接，使用BTS将数据备份到其他的BM上
BM进行读操作时，本地数据先从DS读内存，MS读磁盘，远程数据先使用CM与其他BM简练连接，然后BTS从其他BM上读取数据

Application的调度机制，调度算法有两种，一种是spreadOutApps（平均分配算法），另外一种是非spreadOutApps（将app分配到尽可能少的worker上），可以在配置文件中配置使用哪个算法，遍历可用的worker，可以被app使用的worker，根据剩余的cpu数量倒序排列，针对选出的worker进行资源分配，创建excutor，启动excutor，设置application的状态为running。

• ShuffleMapTask

TaskRunner入口，对序列化的数据进行反序列化得到需要拷贝的文件、jar及task内容，拷贝需要的文件、jar包等（获取hadoop的配置文件），通过task字节码及当前线程的类加载器反序列化为task。开始执行task.run()，创建taskContext即task执行的上下文，调用抽象方法runTask（子类ShuffleMapTask、ResultTask），使用ShuffleMapTask的runTask方法，获取ShuffleManager，进行rdd.iterator方法计算任务算子，对结果使用ShuffleManger进行分区后写入磁盘，返回MapStatus到Task .run()，如果下一个还是ShuffleMapTask，就会去联系MapoutputTasktracker获取上一个的结果，ResultTask也是一样的。然后对MapStatus进行序列化及封装，计算task的统计信息，调用execbackend更行task状态

Spark的spark-submit脚本详解

需要将spark需要的第三方jar打入spark工程jar，将spark和hadoop的依赖配置成provided，仅仅在开发和编译时有效，打包时不会打入这些jar，集群管理器会提供这些jar。
spark-submit封装了spark-class，—verbose打印详细的配置及调试信息

Spark应用提交模式

  • local模式
  • standalone模式
  • yarn-client模式
  • yarn-cluster模式

    Spark核心组件

  • Driver（进程），我们编写的spark程序就是在该进程中运行的
  • Master（进程），负责资源的调度和分配，还有集群的监控等职责
  • Worker（进程），用自己的内存存储RDD，启动其他进程或线程对RDD上的partition进行计算
  • Executor（进程），对RDD中的partiton进行执行
  • Task（线程），并发执行partition中的计算任务

    Spark的local模式提交任务

    local模式的基本原理

    没有所谓的master和worker的概念，相当于启动一个本地进程，然后在一个进程内，模拟spark集群中运行作业，一个spark作业就对应了一个或者多个executor进程。

    Spark的standalone-client模式提交任务

    —master spark：//host_ip:port
    默认就是clinet模式，会在后台启动进程如下：
    SparkSubmit进程，Driver进程
    CoarseGrainedExecutorBackend进程，Executor进程
    Spark的standalone模式，即是基于Spark的Master-Worker集群。spark基于standalone模式的任务提交过程如下图：

具体提交步骤如下：
1. spark-submit脚本通过反射的方式，创建和构造一个DriverActor进程，即Driver进行
2. 进行SparkContext的初始化过程，创建DagScheduler和TaskSchedulerImpl。DagScehduler主要用于将提交的job划分为多个Stage，将Stage封装成Taskset提交给TaskScheduler。TaskScheduler主要负责把每个TaskSet里的task提交到Executor执行。
3. TaskScheduler创建SparkDeploySchedulerBackend与Master进行通讯，请求Master进行Application的注册和资源调度（资源调度算法）
4. Master根据Driver的请求，在各个Worker上进行资源调度，即划分资源运行Executor。
5. Worker进行启动Executor的执行。
6. Executor启动后，反向注册到Driver的TaskScheduler。结束SparkContext初始化，继续执行自己编写的代码。
7. 每执行到一个Action操作时就会创建一个job，然后提交给DAGScheduler。DagScehduler主要将提交的job划分为多个Stage，将Stage封装成Taskset提交给TaskScheduler。
8. TaskScheduler主要负责把每个TaskSet里的task提交到Executor执行。
9. Executor每接受一个task，会封装成TaskRunner，然后从线程池中取出一个线程，执行这个task。
10. TaskRunner将我们编写的代码，也就是执行的算子以及函数拷贝，反序列化，然后执行task。 Task有两种，ShuffleMapTask和ResultTask，只有最后一个Stage的task是ResultTask，之前的Stage都是ShuffleMapTask，
总结：spark程序的执行就是stage分批次作为taskset提交到executor执行，每个task针对Rdd的一个Partition执行定义的算子和函数

standalone模式下的多作业资源调度

该模式下默认是FIFO策略，对多个作业同时执行的支持并不好，因为先提交上来的每一个作业都会尝试使用集群中所有的可用cpu资源，可以通过spark.cores.max进行限制每个作业的最大cpu数量， 建议使用—conf的方式，在spark-submit中进行配置，避免在sparkConf中使用硬编码进行设置

standalone模式下的作业监控和日志记录

通过spark的web ui进行监控，作业运行完毕后就看不到了，这个跟history server有关，在每个算子中打印出来的信息位于对应的worker节点的工作目录中，默认是位于Spark_home/work目录下，在web中也是可以看到的。

yarn-client VS yarn-cluster

•    yarn-client用于测试，因为driver运行在本地客户端，负责调度Application，会与yarn集群产生超大量的网络通讯，导致网卡流量激增，好处是本地可以看到log，便于调试。 
•    yarn-cluster用于生产，因为driver运行在随机的NM上，不存在网卡流量激增的模式。缺点是看不到日志。只能通过yarn application来查看
•    yarn-client模式下AM（executorluncher）只是负责申请资源，然后启动executor，任务调度还是driver，yarn-cluster模式下AM和driver是一起的

Spark基础—RDD

Posted on 2020-03-11 | Post modified: 2020-03-14 | In bigdata

RDD的概念

弹性分布式数据集，抽象的数据集合；含有多个分区，分区散落在不同的节点上，可以并发执行；内存优先存放，若内存放不下时，会缓存到磁盘；具有容错性。

RDD的创建

集合创建，本地文件创建，HDFS文件创建（spark默认会为hdfs的每个block创建一个partition，手动指定时只能比block多

RDD的操作

操作分为transformation和action，transformation特点是lazy特性，在没遇到action操作时不会真正执行，在执行到action时，Driver进程才会提交大量的task到注册过的woker上去。

RDD的持久化策略

多种持久化策略，选择在于平衡CPU和内存消耗之间的平衡:

  • MEMERY_LONLY，以非序列化方式存储与内存中，占用空间大，无CPU消耗
  • MEMERY_ONLY_SER 以序列化方式存储于内存中，占用空间小，消耗内存
    如需进行快速失败恢复，建议使用后缀为_2的策略。尽量不要使用DISK相关的策略，有时候还不如重新运行一次

    RDD的宽依赖与窄依赖

  • 窄依赖，一个Rdd，对它的父RDD，只是简单的一对一关系
  • 宽依赖，每一个父RDD的partition中的数据，都可能传输一部分到下一个Rdd中的每个partition中，此时父子RDD就会出现错综复杂的关系，就是shuffle

Java算法—布隆过滤器

Posted on 2020-03-11 | Post modified: 2020-03-14 | In java

什么是布隆过滤器

布隆过滤器（Bloom Filter）是一个叫做 Bloom 的老哥于1970年提出的。我们可以把它看作由二进制向量（或者说位数组）和一系列随机映射函数（哈希函数）两部分组成的数据结构。相比于我们平时常用的的 List、Map 、Set 等数据结构，它占用空间更少并且效率更高，但是缺点是其返回的结果是概率性的，而不是非常准确的。理论情况下添加到集合中的元素越多，误报的可能性就越大。并且，存放在布隆过滤器的数据不容易删除。

位数组中的每个元素都只占用 1 bit ，并且每个元素只能是 0 或者 1。这样申请一个 100w 个元素的位数组只占用 1000000 / 8 = 125000 B = 15625 byte ≈ 15.3kb 的空间。
总结：一个名叫 Bloom 的人提出了一种来检索元素是否在给定大集合中的数据结构，这种数据结构是高效且性能很好的，但缺点是具有一定的错误识别率和删除难度。并且，理论情况下，添加到集合中的元素越多，误报的可能性就越大。

布隆过滤器原理

当一个元素加入布隆过滤器中的时候，会进行如下操作：

  1. 使用布隆过滤器中的哈希函数对元素值进行计算，得到哈希值（有几个哈希函数得到几个哈希值）。
  2. 根据得到的哈希值，在位数组中把对应下标的值置为 1。
    当我们需要判断一个元素是否存在于布隆过滤器的时候，会进行如下操作：
  3. 对给定元素再次进行相同的哈希计算；
  4. 得到值之后判断位数组中的每个元素是否都为 1，如果值都为 1，那么说明这个值在布隆过滤器中，如果存在一个值不为 1，说明该元素不在布隆过滤器中。
    不同的字符串可能哈希出来的位置相同，这种情况我们可以适当增加位数组大小或者调整我们的哈希函数。
    综上，我们可以得出：布隆过滤器说某个元素存在，小概率会误判。布隆过滤器说某个元素不在，那么这个元素一定不在。

    布隆过滤器使用场景

  5. 判断给定数据是否存在：比如判断一个数字是否在于包含大量数字的数字集中（数字集很大，5亿以上！）、 防止缓存穿透（判断请求的数据是否有效避免直接绕过缓存请求数据库）等等、邮箱的垃圾邮件过滤、黑名单功能等等。
  6. 去重：比如爬给定网址的时候对已经爬取过的 URL 去重

    Redis v4.0 之后有了 Module（模块/插件） 功能，Redis Modules 让 Redis 可以使用外部模块扩展其功能 。布隆过滤器就是其中的 Module。

    常用命令一览

    注意： key:布隆过滤器的名称，item : 添加的元素。
    1. BF.ADD：将元素添加到布隆过滤器中，如果该过滤器尚不存在，则创建该过滤器。格式：BF.ADD {key} {item}。
    2. BF.MADD : 将一个或多个元素添加到“布隆过滤器”中，并创建一个尚不存在的过滤器。该命令的操作方式BF.ADD与之相同，只不过它允许多个输入并返回多个值。格式：BF.MADD{key} {item} [item …] 。
    3. BF.EXISTS : 确定元素是否在布隆过滤器中存在。格式：BF.EXISTS {key} {item}。
    4. BF.MEXISTS：确定一个或者多个元素是否在布隆过滤器中存在格式：BF.MEXISTS {key} {item} [item …]。
      另外，BF.RESERVE 命令需要单独介绍一下：
      BF.RESERVE {key} {error_rate} {capacity} [EXPANSION expansion]。
      下面简单介绍一下每个参数的具体含义：
    5. key：布隆过滤器的名称
    6. error_rate :误报的期望概率。这应该是介于0到1之间的十进制值。例如，对于期望的误报率0.1％（1000中为1），error_rate应该设置为0.001。该数字越接近零，则每个项目的内存消耗越大，并且每个操作的CPU使用率越高。
    7. capacity: 过滤器的容量。当实际存储的元素个数超过这个值之后，性能将开始下降。实际的降级将取决于超出限制的程度。随着过滤器元素数量呈指数增长，性能将线性下降。
      可选参数：
      ▪ expansion：如果创建了一个新的子过滤器，则其大小将是当前过滤器的大小乘以expansion。默认扩展值为2。这意味着每个后续子过滤器将是前一个子过滤器的两倍。
