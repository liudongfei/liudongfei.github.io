hadoop核心技术（二）—hdfs元数据管理剖析

Posted on 2019-05-14 | Post modified: 2020-03-11 | In bigdata

##概述
Hdfs的元数据存储在namenode的内存中，为了防止namenode宕机导致所有的元数据丢失，需要定期备份内存中的数据到硬盘，但如果元数据内容过大，则在dump时会耗费很长时间并且会影响hdfs的性能，因此出现了secondnamenode的角色。二者配合进行元数据管理。
##元数据管理
元数据管理流程图：

流程：

  1. Client端跟nn（namenode）交互会造成nn元数据的更新，同时基于WAL思想，所有涉及到更新元数据的操作都会写入编辑日志中（edits-in-progress文件）。
  2. snn（secondarynamenode）的进程会定期向nn的进程发起合并edits和fsimage的请求。
    1. nn根据时间和edits文件大小进行判断是否需要合并
    2. 如果需要进行合并则先触发edits文件（图中的edits-in-progress-0009-000X）的回滚操作，生成需要合并的edits文件（edits-0009-0012）和新的edits文件（edits-in-progress-0013-000X）
    3. nn将需要合并的edits文件（edits-0009-0012）发送到snn
    4. snn将上一次dump的fsimage（fsimage-0008）加载到内存中，然后将收到的edits文件（edits-0009-0012）依次加载到内存中
    5. snn将内存中的元数据dump成新的fsimage（fsimage-0012）写入磁盘
    6. Snn将新的fsimage发送给nn，nn接收成功后清除旧的fsimage和edits文件

hadoop核心技术（一）—hdfs读写流程

Posted on 2019-05-14 | Post modified: 2020-03-11 | In bigdata

hdfs写数据文件流程

hdfs写入数据流程图如下：

流程：

  1. client连接namenode存数据文件。
  2. namenode记录一条数据位置信息（元数据，一个元数据大概150b），告诉client存哪里。
  3. client用hdfs的api将数据块（默认128M）存储到datanode上（选择最近的一个datanode建立网络连接）。
  4. datanode将数据水平备份（客户端决定该文件的副本数量）。并且备份完成反馈给client。
  5. client通知namenode存储块完毕。
  6. namenode将元数据同步到内存中。
  7. 另一块重复上面的过程。

Idea常用快捷键（Mac）

Posted on 2019-05-12 | Post modified: 2019-05-13 | In util
快捷键 作用
command + d 快速复制并在下一行插入当前行
command + / 注释当前行的代码
command + y 删除当前行的代码
command + p 方法参数提示
command + j 快速自动生成一些自动代码，如for循环
command + b 快速定位到变量的来源
control + h 查看类的继承关系
command + o 根据类名全局查看类
command + shift + 8 列编辑模式
command + shift + ↑↓ 将选中的代码上下移动
command + option + l 格式化代码
command + option + t 把选中的代码放到try{}if{}里
command + option + b 导航到一个抽象方法的实现代码
shift + enter 切换到下一行
option + enter 快速导入包
control + enter 新建界面或getter，setter界面
control + o 快速覆写父类中的方法
