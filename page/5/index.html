Hadoop核心—NM和DN启动

Posted on 2020-03-14 | Post modified: 2020-03-14 | In bigdata 
Hdfs的设计理念，一次写入，多次读取

Namenode启动
在nn首次启动之前，需要进行nn的format命令，目的是生成初始化的fsimage，用于nn首次启动时进行读取。启动过程如下：
1. nn从本地磁盘读取fsimage和edits文件（首次启动不需要读取edits文件）
2. nn回滚edits文件生成新的edits文件，dump最新的fsimage文件
3. Nn进入安全模式，此时不能接受改变hdfs的命名空间的请求，等待dn进行数据块的报告，检查块的熟练达到指定的阈值时（99.999%），退出安全模式
4. 接收客户端请求，进行对外数据读写服务和集群内部数据的副本维护任务（周期性的接受DN的心跳和块报告）
Datanode启动
dn的启动过程如下：
1. 首先向nn进行服务注册register
2. 向nn进行本节点的数据块的报告
3. 周期性（默认1h）向nn上汇报所有的块信息，心跳是3s一次，心跳带回的结果是nn给dn的命令，如复制或删除某个块，超过10min没有收到dn的心跳，则该节点不可用
4. 对外提供数据文件内容的读写请求

Hadoop核心—HDFS高可用

Posted on 2020-03-14 | Post modified: 2020-03-14 | In bigdata

目标
为了解决NameNode的单点故障（SPOF）问题。
HA的核心
1) 如何保持两个NameNode内存中元数据内容的一致
2) active写edits文件，standby读取edits文件
3) 如何保持edits文件的安全，将edits写入2n+1个JournealNode（进程）中
4) 所有的datanode节点都要向两个NN发送心跳和块报告
5) 客户端访问NN通过proxy进行，由proxy判断哪个NN
HA的配置要点（hdfs-site.xml,具体步骤见官网 ）
1) edits文件共享
2) 两个NN
3) client访问proxy
4) fence（隔离，同一时刻只有一个NN对外服务） ，使用的sshfence方法，要求两个NN之间能够免密ssh登录
HA的故障自动转移
1) 依赖于ZK
2) 两个NN启动后都是standby，需要选举出一个为active
3) 两个NN的监控，ZKFC进程（就是zk的client）
自动故障转移配置
1) 开启故障自动转移，hdfs-site.xml
2) 指定zk集群，core-site.xml

Nginx的基本概述

Nginx是一款轻量级的Web 服务器/反向代理服务器及电子邮件(IMAP/POP3)代理服务器，并在一个BSD-like 协议下发行。由俄罗斯的程序设计师Igor Sysoev所开发，供俄国大型的入口网站及搜索引擎Rambler使用。其特点是占有内存少，并发能力强，事实上nginx的并发能力确实在同类型的网页服务器中表现较好。

Nginx 是一个高性能的 Web 和反向代理服务器, 它具有有很多非常优越的特性:

  • 作为 Web 服务器：相比 Apache，Nginx 使用更少的资源，支持更多的并发连接，体现更高的效率，这点使 Nginx 尤其受到虚拟主机提供商的欢迎。能够支持高达 50,000 个并发连接数的响应，感谢 Nginx 为我们选择了 epoll and kqueue 作为开发模型.
  • 作为负载均衡服务器：Nginx 既可以在内部直接支持 Rails 和 PHP，也可以支持作为 HTTP代理服务器 对外进行服务。Nginx 用 C 编写, 不论是系统资源开销还是 CPU 使用效率都比 Perlbal 要好的多。
