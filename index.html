Redis核心—优化及性能提升

Posted on 2020-03-14 | Post modified: 2020-03-14 | In database

Redis降低内存占用
Redis为列表、集合、散列和有序集合提供了一组配置选项，可以让Redis以更节约空间的方式存储长度较短的结构。
压缩列表是列表、散列和有序集合这3种不同类型的对象的一种非结构化表示，压缩列表以序列化的方式存储数据，这些数据每次被读取的时候都要进行解码，每次被写入的时候也要进行局部的重新编码。
如果集合中包含的所有元素都可以被解释为十进制整数，且这些整数又在平台的有符号整数范围内，同时元素个数又足够少的情况下，Redis就会以有序整数数组的方式存储集合，这种存储方式被称为整数集合。
使用压缩列表编码的配置项：
列表：
list-max-ziplist-entries 512 列表为压缩列表情况下允许包含的最大元素数量512
list-max-ziplist-value 64 列表为压缩列表情况下每个元素的最大体积多少字节
散列：
hash-max-ziplist-entries 512 散列为压缩列表情况下允许包含的最大元素数量512
hash-max-ziplist-value 64 散列为压缩列表情况下每个元素的最大体积多少字节
有序集合：
zset-max-ziplist-entries 512 有序集合为压缩列表情况下允许包含的最大元素数量512
zset-max-ziplist-value 64 有序集合为压缩列表情况下每个元素的最大体积多少字节
集合：
set-max-intset-entries 512 集合为整数集合情况下允许包含的最大元素数量512
长压缩列表和大整数集合带来的性能问题：
1. 随着压缩列表和整数集合的体积越来越大，操作这些结构的速度就会变得越来越慢，此时可用分片结构来解决。
2. 当一个结构突破了压缩列表和整数集合设置的限制条件时，Redis就会自动将它转化为典型的底层结构类型。
分片结构
分片的本质就是将简单的规则将数据划分为更小的部分，然后根据数据的属性确定将数据发送到那个部分。
分片式列表：
在不使用lua脚本的情况下，很难对列表进行分片。
分片式散列：
根据散列中存储的键值对的键进行一定的规则转换，确定将该键存储到哪个分片中。
分片式集合：
根据集合中的元素本身进行一定的规则转换，确定将该元素存储到哪个分片中。
Redis的性能提升
读性能提升
• 保持压缩列表的长度不会太大导致影响性能
• 根据需要选择最好性能的存储结构
• 将大体积对象存储到redis前，考虑压缩
• 使用非事务批量提交的方式执行命令
• 添加只读从服务器
写性能提升
• 检查程序，尽可能减少程序需要读取的数据量
• 使用锁去替换watch类型的事务
• 使用多实例redis，提前进行预分区

Redis核心—主从复制

Posted on 2020-03-14 | Post modified: 2020-03-14 | In database

Redis的主从复制
主从复制配置
主：dir 快照和AOF文件的存储位置
主：dbfilename 快照文件的名称
从：slaveof host port 开启从服务器配置
主从复制步骤：
1. 从：连接主服务器，发送SYNC命令
2. 主：执行BGSAVE命令，并使用缓冲区记录BGSAVE之后执行的所有命令
3. 从：根据配置选项来决定是否继续使用现有的数据处理客户端请求，还是返回错误
4. 主：完成BGSAVE命令，向从服务器发送快照，发送期间继续使用缓冲区记录命令
5. 从：丢弃所有就数据，开始载入最新的快照文件
6. 主：继续向从服务器发送缓冲区里面的命令
7. 从：完成快照文件载入，开始接收命令请求
8. 主：完成缓冲区里命令的发送，从现在开始每执行一个写命令都要向从服务器发送相同的写命令
主从复制的问题：
1. 实际中，最好让主服务器只使用50%～65%的内存，留下30%～45%的内存用于执行BGSAVE命令和创建记录写命令的缓冲区。
2. Redis不支持主主复制
3. 当多个从服务器同时连接主服务器时，会导致主服务器的带宽占用过高，从而出现性能问题
4. 从服务器同样可以拥有自己的从服务器，从而形成主从链
5. 树形结构的主从链是合理的

Redis核心—数据持久化

Posted on 2020-03-14 | Post modified: 2020-03-14 | In database

Redis的数据持久化
数据持久化的两种方式：
1. 快照（snapshotting），可以讲存在于某一时刻的所有数据写入硬盘里。
2. 只追加文件（append-only file），在执行写命令时，讲被执行的写命令复制到硬盘里。
两种方式可以单独使用，也可以同时使用。
快照持久化
快照持久化配置项：
save 60 1000 多久执行一次快照操作，60秒内有1000次写入
stop-writes-on-bgsave-error no 创建快照失败后是否继续执行写命令
rdbcompression yes 是否对快照文件进行压缩
dbfilename dump.rdb 硬盘上的快照文件名
快照的检查命令：
redis-check-dump 检查快照文件的正确性
创建快照的方式：
1. 客户端使用BGSAVE命令创建一个快照。Redis调用fork创建一个子进程来负责将快照写入磁盘，主进程继续处理命令请求。
2. 客户端使用SAVE命令创建一个快照。Redis接收到SAVE命令之后，在快照创建完成之前不再响应任何其他命令。
3. 用户设置的save配置选项被触发，会调用BGSAVE命令创建一个快照。
4. 服务器接收到SHUTDOWN命令时，会触发一次SAVE命令创建一个快照。
5. 当一个Redis服务器连接另一个Redis服务器，并向对方发送SYNC命令来开始一次复制操作的时候，主服务器会执行一次BGSAVE命令
快照持久化的问题：
1. 快照持久化只适用于那些即使丢失一部分数据也不会造成任何问题的应用程序。
2. 随着Redis占用的内存越来越多，BGSAVE命令的耗时也会越来越长，此时如果系统的剩余内存不多时，会引发系统大量的使用虚拟内存，从而导致Redis的的性能降低至无法使用。
3. 为了防止Redis创建子进程而导致停顿，建议关闭自动保存，通过手动执行SAVE或BGSAVE命令进行持久化。
AOF持久化
AOF持久化配置项：
appendonly no 是否使用AOF持久化
appendfsync everysec [always] [no] 多久才将写入的内容同步到磁盘，每秒
no-appendfsync-on-rewrite no 在对AOF进行重写的时候能否执行同步操作
auto-aof-rewrite-percentage 100 多久执行一次AOF文件重写
auto-aof-rewrite-min-size 64mb 当AOF文件大于64M，并且AOF文件的体积比上一次重写后增加一倍
AOF的检查和修复命令：
redis-check-aof —-fix 检查并修复AOF文件
AOF持久化的问题：
1. AOF文件越来越大，会造成Redis重启后的还原操作耗时越来越长，解决的方式是进行定期执行AOF文件重写。
AOF文件重写：
1. 为了解决AOF文件持续增大的问题，可以使用BGREWRITEAOF命令来移除AOF文件中的冗余命令来重写AOF文件，使AOF文件变小
2. BGREWRITEAOF命令和BGSAVE命令的原理相同，因此也存在相同的问题。
3. 通过配置项自动触发执行BGREWRITEAOF命令。

Redis核心—事务

Posted on 2020-03-14 | Post modified: 2020-03-14 | In database

Redis的事务机制
事务相关的命令：
discard 取消事务
exec 执行事务
multi 开启事务
unwatch 取消watch命令对于所有key的监控
watch 监控一个key，如果事务执行之前这个key被其他命令改动，那么事务会被打断
Redis的事务机制一次可以执行多个命令：
1. 使用watch命令对键进行指定的监控
2. 使用multi开启事务
3. 将要执行的命令放入缓存队列
4. 在watch监控后exec提交前这段时间，如果被监控的键发生改变，则使用discard取消事务，或者重试操作
5. 如果被监控的键没有发生改变，使用exec提交事务执行命令
Redis事务的特点：
1. 单个redis的命令的执行是原子性的，但redis在事务上没有增加任何维持原子性的机制，所以事务的执行并不是原子性的
2. 事务可以理解为一个打包批量执行的脚本，但批量命令并非原子化操作，中间某条命令失败不会导致前面已做的指令回滚，也不会造成后续的指令不做
3. 这个事务可以在不被其他客户端打断的情况下执行多个命令
4. Redis只会在数据已经被其他客户端抢先改变了的情况下，通知执行了watch命令的客户端，这种做法被称为乐观锁。关系型数据库则使用的是悲观锁
5. 对于上述事务去掉watch命令，可以变成非事务流水线，可以尽可能的减少应用程序和Redis之间的通讯往返次数
6. 随着应用负载的增加，由watch命令导致的程序进行重试的次数越来越多，导致资源浪费。因此，应减少对watch的使用。
Redis使用SETNX构建锁：
1. 使用setnx命令获取锁（包含锁的超时时间）。这个命令只会在键不存在的情况下为键设定值。
2. 执行事务内容
3. 检查是否还持有该锁，并释放锁

Redis基础—基本概述

Redis简介
· 异常快速：每秒能执行11万集合，每秒约81000+条记录
• 支持丰富的数据类型：支持字符串、列表、集合、有序集合、散列数据类型
• 操作都是原子性：所有的redis操作都是原子的
• 支持多个数据库，并且每个数据库中的数据是隔离的，不可共享，并且基于单机才有，集群模式没有数据库的概念。数据库默认从0开始以递增数字命名，默认支持16个，可以修改配置支持更多，客户端连接时自动连接数据库0，可以使用select命令选择数据库
• 不支持自定义数据库的名称，不支持为每个数据库设置不同的密码。FlushAll可以清空一个Redis实例中的所有数据
Redis基本数据类型
字符串（String）
该类型是一个由字节组成的序列，底层存储为二进制，可以包含任何数据，最大存储512M。
该类型明确区分字符串、整数、浮点数，对于整数、浮点数可以执行自增或自减操作。
该类型对于不存在的键进行自增或自减操作，初始化值默认为 0。
列表（List）
该类型是一种双向链表，可以从列表的两端推入或者弹出元素。
集合 （Set）
该类型以无序的方式存储多个各不相同的元素。
有序集合（Zset）
该类型给Set中的元素赋予一个权重（分数）映射，按照权重排序。
散列（Hash）
Hash：散列，键值对集合（字典），适合存储对象，可以update某一属性。

共享变量
默认情况下，算子的函数内使用到的外部变量，都会被拷贝一份副本到每一个task中，如果某个变量特别大就会占用网络传输和内存空间，如果使用共享变量，则变量在每个节点只有一个副本，各个task共享这个变量。
累加器的值只能在Driver中取出
CheckPoint
Checkpoint是spark提供的比较高级的功能，比如在比较复杂的app中，某个rdd要经常用到，某个任务经常失败，重试时需要父RDD的数据，此时数据由于数据丢失需要重新进行计算，此时需要用到checkpoint，将需要的rdd持久化到一个高可用文件系统中。
如何进行checkpoint
SparkContext.setCheckpointDir()
Rdd.checkpoint()
checkpoint原理
一个RDD如果要checkpoint，则会在该RDD所在的job结束后，单独起一个job沿着当前job的lineage向上查找每个需要checkpoint的RDD，将数据写入指定的文件系统路径中，写入成功后，会将该rdd的所有依赖全部删除，将父Rdd设置为CheckpointRDD。 将一个RDDcheckpoint时，要先设置该RDD的持久化persist到磁盘，如果没有persist那么后面进行checkpoint时需要重新计算一次rdd。
checkpoint与persist的区别
与持久化的区别，持久化只是将数据保存BM中，依赖关系时不变的，但是RDD的依赖改变了，其次是持久化的数据是可能丢失的，但是checkpoint的数据丢失的可能性非常低。

Spark基础—DF

Spark-sql的DataFrame
SparkSQL的核心编程抽象，以列的形式组织的，分布式的数据集合 。SQLContext，SparkSQL的上下文。HiveContext是SQLContext的子类，包含除了SQLContext之外的一些针对Hive的功能，支持使用UDF，对与Spark1.3以上的版本都推荐使用HiveContext 。
DataFrame的生成
⁃ 使用反射的方式生成
javaAPI仅支持简单的javabean方式，row形式的RDD
注：java的row在从DF转换成RDD时，会发生数据列顺序的变化
scalaAPI支持复杂的javabean方式，row形式的RDD，需要使用SQLContext的隐式转换
注：scala的row在从DF转换成RDD时，会保持数据列的顺序
⁃ 使用schema的方式编程动态生成
JavaAPI使用RowFactory生成Row，使用DataTypes生成StructField和StructType
ScalaAPI，Row，StructField和StructType都是caseclass
Spark-sql读取parquet文件
parquet是面向分析型业务的列式存储格式，对比行式存储的优势有：
▪ 可以跳过不符合条件的列，只读取需要的数据，降低磁盘的IO
▪ 压缩编码可以降低磁盘存储空间，同一列的数据类型相同
▪ 只读取需要的列，支持向量运算，能获取更好的扫描性能
Sparksql对parquet数据源的支持在如下几个方面：
✓ 支持parquet数据源的自动分区推断，可以根据目录名自动推断出分区信息。目前仅支持推断数值类型和字符串类型的，通过设置如下参数spark.sql.sources.partitionColumnTypeInference.enabled，默认为true，设置为false即是禁用推断分区列的数据类型，此时所有分区列的数据类型都为String
✓ 支持多个parquet文件元数据合并，因该操作比较消耗性能，从1.5开始默认是关闭的读取parquet数据源时，将选项mergeSchema，设置为true。使用SQLContext.setConf()，将spark.sqlparquet.mergeSchema参数设置为True。通过上面两种方式中任意一种开启
Spark-sql对hive进行数据的读写
操作hive时要使用HiveContext，继承于SQLContext
dataset和dataframe的区别
dataframe就是dataset[row],是untyped
dataset是dataset[T],是typed类型的
dataset和dataframe的相互转换：as、toDF
ds、df基本的action操作和rdd的操作相同，
ds、df基本操作，创建临时视图，获取执行计划
typed操作（需要将df转换成ds）
⁃ coalesce：重新定义分区，只能减少分区，可以选择不发生shuffle操作
⁃ repartition：重新定义分区，可以增加或减少分区，必须进行shuffle
⁃ distinct：去重，一条记录完整内容去重
⁃ dropDuplicates：去重，根据选择的字段内容去重
⁃ except：获取当前ds中存在，但是在另外一ds中不存在的记录
⁃ filter：根据自己的逻辑进行过滤
